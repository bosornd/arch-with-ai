# 용어 사전 (Glossary)

구조 설계 활동에서 사용되는 주요 용어의 정의입니다. 모든 에이전트는 이 용어를 일관되게 사용해야 합니다.

## A

### Architectural Decision (구조적 의사결정)
중요한 품질 요구사항에 영향을 미치는 중요한 설계 결정. 최적 설계에 큰 영향을 미치는 결정을 의미하며, 모든 설계 결정이 구조적 의사결정은 아니다.

### Architecture (아키텍처, 구조)
시스템의 근본적인 조직 구성. 컴포넌트, 컴포넌트 간 관계, 환경과의 관계, 그리고 설계와 진화를 지배하는 원칙으로 구성됨.

## C

### Candidate Architecture (후보 구조)
아직 채택 여부가 결정되지 않은 구조적 의사결정. 평가를 통해 채택 또는 기각됨.

### Component (컴포넌트)
시스템을 구성하는 구조적 단위. 명확한 책임과 인터페이스를 가지며, 다른 컴포넌트와 상호작용함.

### Conflicting Candidates (상충하는 후보 구조)
동시에 적용할 수 없는 후보 구조들. 서로 다른 접근 방식을 제안하는 후보들로, 비교 평가를 통해 하나를 선택해야 함.

### Connector (커넥터)
컴포넌트 간의 상호작용 메커니즘. 데이터 흐름, 제어 흐름, 통신 프로토콜 등을 정의함.

## D

### Deployment Architecture (실행 동작 구조)
런타임 관점에서 본 시스템 구조. 실행 프로세스, 노드, 네트워크 구성 등을 포함함.

### Domain Model (도메인 모델)
기능적 요구사항을 분석하여 도출된 기능적이고 개념적인 구조. 핵심 도메인 개념과 그들 간의 관계를 표현함.

## F

### Final Architecture (최종 구조)
채택된 후보 구조들을 통합하여 설계된 최종 시스템 구조. 실행 동작 구조와 모듈 구조를 모두 포함함.

### Functional Requirement (기능적 요구사항)
시스템이 수행해야 하는 기능과 동작을 정의하는 요구사항. Use Case로 표현됨.

## M

### Module Architecture (모듈 구조)
개발 관점에서 본 시스템 구조. 코드 모듈, 패키지, 레이어, 의존성 등을 포함함.

## N

### Non-Functional Requirement (NFR, 비기능적 요구사항)
반드시 만족해야 하는 품질 요구사항. 측정 가능한 허용치가 명확하며, 만족하지 못하면 과제가 실패로 판정됨.

**특징**:
- 명확한 측정 기준과 허용치 필요
- 미충족 시 시장 진입 불가
- 허용치 이상으로 만족하는 것은 요구하지 않음

**예시**: `[부팅 시간] < 2초`, `[가용성] >= 99.9%`

## O

### Optimal Design (최적 설계)
비즈니스 전반에 걸쳐 최고의 품질 수준을 달성하는 설계. 모든 비기능적 요구사항을 만족하면서 우선순위가 높은 품질 속성을 더 많이 만족하는 설계.

## Q

### Quality Attribute (QA, 품질 속성)
더 많이 만족할수록 좋은 품질 요구사항. 시스템의 평가 요인이자 시장 경쟁력의 척도. 우선순위가 중요하며, 명확한 허용치는 없음.

**특징**:
- 시장 경쟁력 평가 기준
- 우선순위 기반 의사결정
- 트레이드오프 허용
- 측정 가능하나 허용치는 명확하지 않음

**예시**: 성능, 변경 용이성, 보안, 가용성, 확장성, 사용성 등

### Quality Requirements (품질 요구사항)
시스템의 품질을 평가하는 요구사항. 비기능적 요구사항(NFR)과 품질 속성(QA) 두 가지 유형으로 구분됨.

### Quality Scenario (품질 시나리오)
품질에 대한 정의.


## U

### Use Case (유스 케이스)
시스템이 사용자에게 제공하는 기능을 표현하는 방법. 액터, 사전조건, 사후조건, 주요 시나리오로 구성됨.

## 명명 규칙

### 문서 파일명

#### Use Case
- 형식: `UC-{번호}-{제목}.md`
- 예시: `UC-001-사용자-등록.md`, `UC-002-주문-처리.md`
- 번호: 3자리 숫자 (001부터 시작)

#### Quality Scenario
- 형식: `QS-{번호}-{제목}.md`
- 예시: `QS-001-사용자-등록-처리-시간.md`, `QS-002-결재-시스템-변경-용이성.md`
- 번호: 3자리 숫자 (001부터 시작)

#### Non-Functional Requirement
- 형식: `NFR-{번호}-{제목}.md`
- 예시: `NFR-001-사용자-등록-처리-시간.md`
- 번호: 3자리 숫자 (001부터 시작)

#### Quality Attribute
- 형식: `QA-{번호}-{제목}.md`
- 예시: `QA-001-사용자-등록-처리-시간.md`, `QA-002-결재-시스템-변경-용이성.md`
- 번호: 3자리 숫자 (001부터 시작), 우선 순위

#### Candidate Architecture
- 형식: `CA-{번호}-{제목}.md`
- 예시: `CA-001-사용자-등록-서비스-분할.md`, `CA-002-결재-시스템-추상화.md`
- 번호: 3자리 숫자 (001부터 시작)

### ID 참조 규칙

문서 내에서 다른 항목을 참조할 때:
- Use Case 참조: `UC-001`
- Quality Scenario 참조: `QS-001`
- Non-Functional Requirement 참조: `NFR-001`
- Quality Attribute 참조: `QA-001`
- Candidate Architecture 참조: `CA-001`

## 약어 목록

- **NFR**: Non-Functional Requirement (비기능적 요구사항)
- **QA**: Quality Attribute (품질 속성)
- **UC**: Use Case (유스 케이스)
- **QS**: Quality Scenario (품질 시나리오)
- **CA**: Candidate Architecture (후보 아키텍처)
- **AD**: Architectural Decision (구조적 의사결정)
